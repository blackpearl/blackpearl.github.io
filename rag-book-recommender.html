// overview

What This Project Does

This project demonstrates how to build a production-ready RAG pipeline entirely on local infrastructure. It ingests book summaries from a web source, stores semantic embeddings in a persistent vector database, and powers a conversational chatbot that retrieves and reasons over that knowledge — all without a single external API call.

The core insight: RAG separates knowledge storage from reasoning. Your documents live in a vector database as embeddings; when a user asks a question, the system retrieves only the most relevant chunks and hands them to the language model as context. The LLM never needs to memorise your data — it just needs to read and reason over what's retrieved.

// industry application

Transforming the Insurance Industry

The same RAG architecture powering this book chatbot is directly applicable to one of the most document-heavy, compliance-sensitive industries in the world: insurance. Insurance companies sit on vast repositories of unstructured knowledge — policy documents, claim guidelines, regulatory filings, product manuals — that are difficult for both customers and internal teams to navigate.

A RAG system turns that static knowledge into a live, queryable intelligence layer — without fine-tuning a model or sending sensitive data to a public API.

Why local RAG matters for insurance: Policy documents, claims data, and customer records are highly sensitive. Running Ollama + Weaviate Embedded keeps everything on-premises — no data leaves the organisation, satisfying GDPR, HIPAA, and internal compliance requirements out of the box.

// roadmap

Future Steps & Improvements

This project is a working foundation. Below are the most impactful directions for making the system more accurate, robust, and production-ready.

filtering

Hybrid BM25 + Vector Search

Combine dense vector search (semantic similarity) with sparse BM25 keyword search. Weaviate supports hybrid search natively — tuning the alpha parameter allows fine-grained control over how much weight to give semantic vs. lexical matching, improving precision on domain-specific terminology.

filtering

Dynamic Tag Vocabulary

Instead of a hardcoded KNOWN_TAGS list, derive tags automatically from the ingested content using clustering or topic modelling. As new content is added, the tag vocabulary evolves — no manual maintenance required.

filtering

Query Rewriting

Before embedding, use the LLM to rewrite the user query into an optimal retrieval form. For example, "something about getting better at work" becomes "books on career development, productivity, and professional growth" — improving vector search recall significantly.

evals

Retrieval Quality Evals

Build an evaluation harness using a ground-truth dataset of (query → expected books) pairs. Measure Recall@K and MRR (Mean Reciprocal Rank) to quantify how well the retrieval step is performing. Run evals on every change to the embedding model or chunking strategy.

evals

Answer Faithfulness Evals

Use an LLM-as-judge approach (or a framework like RAGAS) to score whether the generated answer is grounded in the retrieved context. Catch hallucinations early — especially critical in regulated industries where a wrong answer has real consequences.

evals

Guardrail Accuracy Evals

The query classifier (is_book_recommendation_query) needs its own eval set of positive and negative examples. Track false positive rate (valid queries incorrectly rejected) and false negative rate (off-topic queries that slip through) separately.

infrastructure

Chunking Strategy

Currently each book is stored as a single vector over its full summary. For longer documents, splitting into overlapping chunks (with a sliding window) dramatically improves retrieval precision — the model retrieves the specific relevant passage, not a diluted whole-document embedding.

infrastructure

Incremental Re-ingestion

Build a scheduled crawler that detects new or updated content and re-ingests only the delta. Combined with the existing wp_post_id idempotency key, this keeps the knowledge base fresh without a full rebuild.

ux

Streaming Responses

Switch the Ollama generation call to streaming mode and pipe tokens to the UI via Server-Sent Events (SSE). Answers appear word-by-word, dramatically improving perceived latency — especially important when llama3.2 is generating long responses on CPU.

The Core RAG Pattern

The entire pipeline reduces to a single composable function. This is the pattern that makes RAG so powerful — and so portable across domains.

def run_rag_query(query: str, db: WeaviateClient) → dict:
  # 1. Guard: is this a valid domain query?
  if not is_valid_query(query): return rejection

  # 2. Extract structured filters from natural language
  filters = extract_filters_semantic(query) # LLM → structured

  # 3. Embed the query into vector space
  vector = embed(query) # same model used at ingest time

  # 4. Retrieve: semantic similarity + metadata filters
  context = db.near_vector(vector, filters=filters, limit=5)

  # 5. Generate: ground the LLM in retrieved context
  return llm.generate(build_prompt(query, context))
